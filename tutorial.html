Chapter 1 · 30 sec 速读 🌊 AFlow 是什么?

AFlow(FoundationAgents/AFlow) 是 MetaGPT 团队 2024 年发的一个框架。一句话: 让 LLM 自动给你设计 prompt + workflow 结构,免去你手写 CoT / ReAct / Self-Consistency 的工作。

⚠️ 它解决的问题 你要给 LLM 做一个任务(比如 multi-hop QA),怎么写 prompt 最有效?手写 CoT 还是 ReAct? 每种方法在不同 task 上效果不一样,而且都是凭直觉拼出来的。不可证明最优。
💡 它的解法 把"agentic workflow"看作一个 DAG of LLM calls(节点 = operator,边 = 数据流)。 搜这个空间。用 MCTS 找局部最优 workflow。

训练完得到的产出:一对 Python 文件 graph.py(workflow DAG 代码)+ prompt.py(workflow 用的 prompt 字符串)。 import 进去就能在新数据上跑这个最优 workflow。

💡 Aha moment AFlow 不发明新的 prompting technique。它只是搜索"如何最优地组合已有的 operator"(CoT、ensemble、refine、verify 等)。 所以它的产出是组合方案,不是新算法。

Chapter 2 · 核心抽象 🧠 心智模型

一个 AFlow workflow,本质上就是这么个 Python class:

class Workflow: def __init__(self, llm_config): self.llm = create_llm_instance(llm_config) # 准备这个 workflow 要用的几个 Operator 实例 self.answer_generate = AnswerGenerate(self.llm) self.sc_ensemble = ScEnsemble(self.llm) self.custom = Custom(self.llm) async def __call__(self, problem: str): # 这里写 workflow 的 DAG 逻辑 result, cost = ..., ... return result, cost

记住这 3 点就够:

  • Workflow 是个 async class,核心是 __call__(problem) 方法
  • 里头通过 await 调一连串 Operator(LLM call 的 reusable 模板)
  • DAG 是隐式的——就是 await 顺序 + 数据依赖。AFlow 没有显式 graph definition
💡 Aha moment 所有 AFlow trained workflow 都长这样。所以"理解 AFlow"约等于"理解几个核心 Operator 怎么工作"。这就是下一章。

Chapter 4 · LEGO 砖块 🧱 核心 Operators

这一章把 4 个最关键的 operator 拆开看。每个都说签名、输入输出、什么时候用。

⚙️ Custom 最通用的 LLM call

任意 prompt + 任意 input,返回 {"response": str}。可以理解成"用我给的 prompt 调一次 LLM"。

async def __call__(self, input: str, instruction: str): # instruction 是 system-level 的 prompt # input 是当前 task 的具体内容 # 返回 {"response": "..."}

典型用例:HotpotQA round_3 trained workflow 用它把 verbose 答案压缩成短 span。instruction 是训练好的 FORMAT_ANSWER_PROMPT。

❓ AnswerGenerate QA 格式回答

给个问题,内置 prompt 让 LLM 输出 thought + answer。返回 {"thought": str, "answer": str}。

async def __call__(self, input: str): # prompt 内置 - 让 LLM 用 <thought>...</thought><answer>...</answer> 格式回答 # 解析后返回 {"thought": "...", "answer": "..."}

典型用例:HotpotQA workflow 里 3 次并行调用,生成 3 个 candidate answer。

🗳️ ScEnsemble 自洽投票

"Self-Consistency Ensemble"。给一组 candidate solutions,让 LLM 投票选最一致那个。返回 {"thought": str, "response": str}。

async def __call__(self, solutions: List[str], problem: str): # 内部 prompt 给 LLM 看 N 个 solutions,让它选一个 # 实际用 letter (A/B/C) 投票,然后回填到 solutions[idx] # 返回 {"thought": "选 A 因为...", "response": solutions[A]}

典型用例:几乎所有 trained workflow 都用。3-5 个 candidate → 1 个 best。

✅ 实验数据 在我们 Qwen2.5-7B 实验中,4 个 benchmark 的 voting error 都 ≤0.5%。这个 operator 在 weak model 上工作得非常好。

🐍 Programmer 生成并执行 Python

让 LLM 写 Python 解题,然后真的跑代码(sandbox 里),拿 stdout 当答案。返回 {"code": str, "output": str}。

async def __call__(self, problem: str, analysis: str = "None"): # 内部 retry 3 次:让 LLM 生成 def solve(): ... return result # 在 ProcessPoolExecutor sandbox 跑,timeout 30s # 失败时返回 {"code": None, "output": "No code generated"}

典型用例:MATH 和 GSM8K workflow 用它做"算术验证"。

⚠️ Weak model 上的失败 Qwen-7B 在我们实验里几乎所有 Programmer 调用都返回 "No code generated"——代码格式要求严,小模型生不出。 这是 weak model 跑 paper workflow 的主要痛点。

还有 Test, Format, Review, Revise, MdEnsemble 等 — 主要给代码任务(HumanEval/MBPP)用。看 scripts/operators.py 全列表。

💡 Aha moment AFlow 的 magic 在于:这些 Operator 都是预定义的(人写死的代码)。 MCTS 不学新 operator,只学"怎么排列、用什么 prompt 配置它们"。这把搜索空间从 ∞ 缩到可行。

Chapter 5 · 解剖一个真实 workflow 🔬 完整流程拆解

看 AFlow 在 HotpotQA 上训出来的 best workflow(round_3):

# data/results/HotpotQA/graphs_test/round_3/graph.py class Workflow: def __init__(self, name, llm_config, dataset): self.llm = create_llm_instance(llm_config) self.custom = operator.Custom(self.llm) self.answer_generate = operator.AnswerGenerate(self.llm) self.sc_ensemble = operator.ScEnsemble(self.llm) async def __call__(self, problem: str): solutions = [] for _ in range(3): # ← 3 次独立尝试 initial_response = await self.answer_generate(input=problem) solutions.append(initial_response['answer']) ensemble_result = await self.sc_ensemble(solutions=solutions) # ← 投票 best_answer = ensemble_result['response'] refined_solution = await self.custom( # ← 格式化 input=f"Question: {problem}\nBest answer: {best_answer}", instruction=prompt_custom.FORMAT_ANSWER_PROMPT ) return refined_solution['response'], self.llm.cost_manager.total_cost

为啥是这个结构?

  • 3x AnswerGenerate:LLM 单次输出有随机性,3 次提高 recall(只要有一次答对了就行)
  • ScEnsemble vote:从 3 个候选选最一致的(self-consistency)
  • Custom + FORMAT_ANSWER_PROMPT:LLM 默认答得 verbose,这步压缩成 F1 友好的短 span

FORMAT prompt 长什么样?

# data/results/HotpotQA/graphs_test/round_3/prompt.py FORMAT_ANSWER_PROMPT = """ Given the question and the best answer, format the final answer to be concise, accurate, and directly addressing the question. Ensure the answer is a clear, brief statement without additional explanation or reasoning. If the answer is a name, profession, or short phrase, provide only that information without forming a complete sentence. For example: - If the answer is a person's name, just provide the name. - If the answer is a profession, state only the profession. - If the answer is a short phrase, give only that phrase. Do not include any prefixes like "The answer is" or "The profession is". Just provide the answer itself. """
💡 Aha moment 这个 prompt 是 AFlow 训出来的——不是人写的! MCTS 搜索过程中,某一轮 LLM 提议了这个 prompt 改进,在 val set 上提分了,就被保留。 Paper 所谓的 "trained workflow" 主要训的就是这种 prompt 字符串。

Chapter 6 · 搜索算法 🔄 训练是怎么进行的

AFlow 用 MCTS-style search 找最优 workflow。流程如下:

F1 / accuracy] F --> G{比当前 best 高?} G -->|Yes| H[更新 best] G -->|No| I[MCTS:
探索 vs 利用] H --> B I --> B classDef seed fill:#f3f4f6,stroke:#6b7280,color:#111827 classDef gen fill:#dbeafe,stroke:#3b82f6,color:#1e3a8a classDef eval fill:#fef3c7,stroke:#f59e0b,color:#78350f classDef best fill:#dcfce7,stroke:#22c55e,color:#14532d class A seed class B,D gen class E,F eval class H best

关键设计

  • Optimizer LLM:用一个更强的 LLM(paper 用 GPT-4o)当"workflow 改造师"。它看当前 workflow 代码 + 性能 + 失败案例,提议一个修改
  • Executor LLM:用一个便宜 LLM(GPT-4o-mini)执行 workflow 来评分。这就是训练时强 LLM 设计 workflow,inference 时弱 LLM 跑 workflow
  • MCTS:不是每轮都改最 best 的。有概率 backtrack 到早期分支再分叉,避免 local optimum

真实训练数据(我们看到的 round 数)

Benchmark 总训练轮数 Final best round 观察
HotpotQA21 roundsround_3早期就收敛
DROP21 roundsround_3同 HotpotQA

💡 Aha moment 最 best 的 round 通常很早——HotpotQA/DROP 第 3 轮、MATH 第 5 轮。 说明 MCTS 快速找到"够好"的结构后就 plateau 了。后面 20 轮基本是在 refine prompt。 这暗示workflow 搜索空间不大,主要价值在 prompt tuning。
💰 训练成本 Paper 说每个 benchmark 训练成本 ~$5(以 GPT-4o-mini 为 executor,GPT-4o 为 optimizer)。 每轮要在 val set(~50-100 query)跑完整 workflow + optimizer 提议 + 评分,总 LLM call 在 1000s 级别。

Chapter 7 · 上手实战 🚀 怎么跑起来

1. 安装 + 准备

git clone https://github.com/FoundationAgents/AFlow cd AFlow pip install -r requirements.txt # 准备 LLM API key cp config/config2.example.yaml config/config2.yaml # 编辑 config2.yaml,填 OpenAI key

2. 训练一个新的 workflow

# 训练 HotpotQA workflow,大约 $5,几小时 python run.py --dataset HotpotQA --sample_size 4 \ --question_type qa --optimized_path data/results

训练完输出 data/results/HotpotQA/graphs/round_N/{graph.py, prompt.py}。

3. 直接用已训练好的 workflow(推荐)

# 用 paper 训好的 workflow 在新数据上 inference from data.results.HotpotQA.graphs_test.round_3.graph import Workflow llm_config = {"model": "gpt-4o-mini", "api_key": "sk-..."} workflow = Workflow("hotpotqa", llm_config, dataset="HotpotQA") answer, cost = await workflow(problem="Who founded SpaceX?")
⚠️ 踩过的坑 data.results.HotpotQA.graphs_test.round_3.graph 这个 import 在新版 AFlow 里不能直接用—— 原 graph.py 引用 metagpt.ext.aflow.*,而那个 namespace 在 AFlow spinoff 时被删了。 我们的做法:把 graph.py 当 spec 看,在自己的 runner 里 reimplement(见 src/aflow_runner_v2.py)。

4. 怎么在 weak model 上跑(我们做的)

主要难点是 AFlow 默认设计给 GPT 系列。换成本地 Qwen2.5-7B 需要:

  • 起一个 OpenAI 兼容 endpoint(我们用 openai_shim_batched.py 包装 transformers)
  • 把 config2.yaml 的 base_url 指向那个 endpoint
  • Reimplement trained graph.py(因为旧 import 路径已失效)

如果想自己改进 workflow

基于我们的 failure analysis,DROP benchmark 的 FORMAT step 在 weak model 上有 5.5% query 出 bug (gold 是人名,FORMAT 输出却是数字)。可以试试:

  • 改 REFINE_ANSWER_PROMPT 让它强制保留 entity-level token
  • 在 FORMAT 前加一道 Review operator,人名/日期/数字分类
  • 直接换更大 model 当 FORMAT executor(因为 FORMAT 是最简单的 task)
💡 Aha moment AFlow 不是黑盒。Trained workflow 是 plain Python,prompt 是 plain string。 你可以手动改任何一个,然后用 BaseBenchmark 测一下 score 变化。 这就是为什么写一个 trace inspector 比读 paper 更有用——直接看哪里失败再改。
